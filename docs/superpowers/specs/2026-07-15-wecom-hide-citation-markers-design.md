# 企业微信隐藏引用标记设计

## 目标

企业微信收到 RAG 回答时，不再显示 `[ID:n]` 或系统已支持的等价数字引用标记，但仍然只发送回答正文实际引用的图片。

该改动只影响企业微信的出站文字。Web 端、数据库中的原始回答、其他 Channel 和引用图片筛选结果保持不变。

## 范围

- 企业微信出站文字隐藏内部引用标记。
- 引用图片继续依据清理前的原始回答筛选。
- 支持现有引用解析器识别的 `[ID:n]`、`[n]`、阿拉伯语数字和波斯语数字形式。
- 删除引用标记后，只清理由此产生的多余水平空格。
- 保留 Markdown 的段落、换行、列表缩进和代码块结构。
- 不修改提示词，不要求大模型改变引用输出格式。
- 不改变 Web 端和其他 Channel 的引用展示。

## 方案比较

### 方案一：Channel 出站层单次遍历（采用）

在 Channel 启动层使用一次引用标记遍历，同时收集引用图片并生成企业微信专用的清理后文字。该方案复用现有引用规则，不增加模型、数据库或网络调用，且不会把 RAG 引用规则耦合到企业微信协议代码中。

### 方案二：企业微信协议层替换文字

在企业微信 `send` 方法中删除引用标记。实现位置直观，但会让协议适配器理解 RAG 引用语法，并可能在 WebSocket 与 Webhook 路径中形成重复逻辑，因此不采用。

### 方案三：通过提示词禁止输出引用标记

让大模型不生成 `[ID:n]`。该方案无法保证模型严格遵守，同时会失去筛选实际引用图片所需的信息，因此不采用。

## 架构设计

### Channel 能力声明

在公共 Channel 基类增加默认关闭的“隐藏引用标记”能力。企业微信 Channel 开启该能力，其他 Channel 使用默认值，避免用硬编码的 Channel 名称分支控制行为。

### 单次遍历处理

将现有引用图片筛选辅助逻辑扩展为一个职责明确的出站准备函数。输入为原始回答和格式化后的 `reference.chunks`，输出为：

1. 清理引用标记后的企业微信文字。
2. 按首次引用顺序排列并去重的 `OutgoingImage` 列表。

处理过程只遍历一次引用标记匹配结果：

1. 将当前匹配之前的原始文本追加到文字缓冲区。
2. 规范化引用编号并校验知识块下标。
3. 当知识块存在有效 `image_id` 且尚未发送时，将其加入图片列表。
4. 不把引用标记本身写入文字缓冲区。
5. 继续处理下一个匹配，最后追加剩余文本。
6. 只清理标记删除位置产生的多余水平空格，不改变换行和 Markdown 缩进。

该处理是本地字符串操作，不调用大模型、数据库、对象存储或企业微信接口。相对于回答生成和图片上传，其耗时可以忽略。

### 数据流

```text
async_chat 原始回答（含 [ID:n]）
  -> structure_answer 保存原始回答和 reference.chunks
  -> 企业微信出站准备函数单次遍历
       -> 清理后的文字
       -> 实际引用的图片列表
  -> OutgoingMessage(text, images)
  -> 企业微信先发送文字，再发送引用图片
```

Web 端和数据库继续使用 `structure_answer` 保存的原始回答，因此仍可展示和解析引用标记。

## 错误与边界处理

- 无引用标记时，文字保持原样，图片列表为空。
- 引用下标非法或越界时，从企业微信文字中隐藏该内部标记，但不产生图片。
- 被引用知识块没有 `image_id` 时，只隐藏文字标记，不发送图片。
- 重复引用同一图片时只发送一次，顺序由首次有效引用决定。
- 连续多个引用标记全部隐藏，不留下重复空格。
- 引用标记位于中文标点、英文标点、行尾或段落之间时，不留下标点前空格。
- 图片读取、上传或发送失败时，仍保持现有容错行为，不影响文字回复和后续图片。

## 测试设计

单元测试覆盖以下行为：

- 企业微信出站文字删除单个和多个 `[ID:n]` 标记。
- `[n]`、阿拉伯语数字和波斯语数字引用使用相同规则。
- 删除标记后不产生中文或英文标点前的多余空格。
- Markdown 段落、换行和列表缩进保持不变。
- 图片仍按照清理前的首次引用顺序提取并去重。
- 无图片、非法下标和越界引用不会产生图片。
- 企业微信以外的 Channel 保持原始文字。
- 企业微信发送顺序仍为文字在前、图片在后。

测试不连接真实企业微信，不访问外部网络或对象存储。

## 验收标准

1. 企业微信用户看到的文字中不包含内部引用标记。
2. 实际引用图片的筛选结果与改动前一致。
3. Web 端、数据库原始回答和其他 Channel 的文字不发生变化。
4. 删除标记后不破坏 Markdown 结构，也不留下明显的多余空格。
5. 不增加任何模型、数据库或网络调用。
6. 相关单元测试和现有企业微信 Channel 测试全部通过。
