# 部门管理员与超级管理员权限隔离设计

## 1. 背景

当前 RAGFlow 已经提供普通业务页面和独立的 Admin 管理页面。Admin 页面目前只允许 `is_superuser` 用户登录，主要用于用户、服务和系统配置管理；知识库、聊天助手、聊天记录和消息通道仍主要在普通业务页面中管理。现有 Admin 角色管理接口尚未实现完整的 RBAC 逻辑。

本期需要把现有 9222 端口对应的 Admin 页面统一建设为管理门户，同时服务超级管理员和部门管理员。超级管理员可以管理所有部门、知识库、聊天助手、聊天记录和消息通道；部门管理员只能管理本部门知识库、只读查看本部门聊天助手，并查看、搜索和导出本部门聊天记录。普通员工不登录管理门户，只通过企业微信使用知识库问答机器人。

本期知识库内容不以高度机密资料为主，因此采用固定角色和部门级隔离，不建设复杂的动态 RBAC、字段脱敏和审批流。但所有权限必须在后端强制执行，不能只依赖前端隐藏。

## 2. 设计目标

1. 复用现有 9222 Admin 页面，不新增第三套管理前端。
2. 将一个部门映射为一个独立的 RAGFlow Tenant/Workspace。
3. 提供固定的 `super_admin` 和 `department_admin` 两级管理角色。
4. 保证部门管理员只能访问本部门知识库、聊天助手和聊天记录。
5. 将聊天助手创建、编辑、删除和企业微信 Channel 管理权限集中到超级管理员。
6. 让超级管理员能够在一个门户中跨部门查看和管理所有业务数据。
7. 对导出、删除、凭证操作和跨部门管理操作保留审计记录。
8. 对现有资源提供可控的部门归属迁移流程。

## 3. 非目标

本期不实现以下能力：

- 自定义角色和动态权限组合。
- 多级部门或部门树。
- 一个部门管理员管理多个部门。
- 一个部门配置多个部门管理员。
- 普通员工管理后台账号。
- 字段级脱敏、导出水印、双人审批和审批流。
- 跨部门共享知识库。
- 部门管理员创建、编辑或删除聊天助手。
- 部门管理员查看或管理消息通道、模型密钥和 API Token。

## 4. 核心方案

### 4.1 部门映射为 Tenant

每个部门对应一个独立 Tenant。知识库、聊天助手和 ChatChannel 继续使用现有 `tenant_id` 表示部门归属，不新增一套与 Tenant 平行的数据隔离体系。

Tenant 增加 `tenant_type` 字段，固定取值为 `personal` 或 `department`，默认值为 `personal`。现有个人 Tenant 保持原状，不自动出现在部门列表中。部门名称使用 Tenant 的 `name`，状态使用现有 Tenant 状态字段。

为兼容 RAGFlow 现有“用户 ID 通常等于其默认 Tenant ID”的假设，新建部门时先创建部门管理员用户，再以该用户 ID 创建部门 Tenant。即使以后更换部门管理员，Tenant ID 也保持不变，只调整用户与 Tenant 的成员关系。

### 4.2 固定角色

本期不依赖尚未实现的通用 `RoleMgr`，采用固定角色策略：

- `super_admin`：沿用现有 `User.is_superuser=True`。
- `department_admin`：非超级管理员用户，且在唯一一个有效部门 Tenant 中具有部门管理员成员角色。

部门管理员登录时必须恰好匹配一个有效的部门 Tenant。不存在部门归属、存在多个部门归属或部门已停用时，登录失败并记录安全日志。

### 4.3 统一管理门户

超级管理员和部门管理员都使用现有 9222 Admin 登录页和页面框架。登录成功后，后端返回当前用户的固定角色和允许访问的部门范围；前端据此生成导航菜单、路由和操作按钮。

前端隐藏仅用于改善体验。所有接口仍必须在后端执行角色、动作和部门范围校验。

### 4.4 集中管理聊天助手和消息通道

聊天助手及企业微信 Channel 只能由超级管理员创建和管理。超级管理员创建资源时必须明确选择目标部门：

- 聊天助手写入目标部门 `tenant_id`，只能选择该部门的知识库。
- ChatChannel 写入目标部门 `tenant_id`，只能连接该部门的聊天助手。
- 部门管理员只能读取本部门聊天助手的非敏感配置，不能修改。
- ChatChannel 列表、配置、Bot ID、Secret、连接日志和操作入口不向部门管理员返回。

### 4.5 集中模型配置

模型和密钥由超级管理员统一管理。新建部门时继承系统默认的聊天模型、向量模型、重排模型和解析器配置，部门管理员只能使用这些默认能力创建知识库，不能查看密钥或选择模型。

部门 Tenant 使用系统托管的模型配置。模型解析层应支持部门 Tenant 读取系统默认模型配置，但不能通过部门管理员接口读取或修改模型凭证。模型调用产生的用量仍按部门 Tenant 记录，供部门维度统计。

## 6. 前端信息架构

### 6.1 超级管理员菜单

- 全局总览
- 部门管理
- 部门管理员管理
- 知识库管理
- 聊天助手
- 聊天记录
- 消息通道
- 模型管理
- 审计日志
- 系统设置

知识库、聊天助手、聊天记录和消息通道页面提供部门筛选器。创建知识库、聊天助手或 Channel 时，超级管理员必须先选择目标部门，页面不默认使用超级管理员自己的 Tenant。

### 6.2 部门管理员菜单

- 部门总览
- 知识库
- 聊天助手（只读）
- 聊天记录
- 个人设置

部门管理员页面不显示部门切换器。消息通道、模型管理、用户管理、API Token、审计日志和系统设置等入口不渲染。聊天助手详情页不显示保存、删除、测试或绑定 Channel 操作。

### 6.3 前端路由保护

Admin 授权布局在加载时取得当前角色和部门范围：

- 未登录用户跳转登录页。
- 角色无权访问的路由展示 403 页面。
- 部门管理员直接输入超级管理员页面地址时不发起业务数据请求。
- 前端请求不得替部门管理员拼装或允许修改 `tenant_id`。

前端路由保护不替代后端鉴权。

## 7. 后端鉴权设计

### 7.1 登录与身份上下文

现有 Admin 登录逻辑从“仅允许 `is_superuser`”调整为“允许有效的管理角色”：

1. 校验账号密码和账号状态。
2. `is_superuser=True` 时建立 `super_admin` 身份。
3. 非超级管理员时查询有效部门 Tenant 成员关系。
4. 恰好属于一个部门且角色为部门管理员时，建立 `department_admin` 身份。
5. 其他用户拒绝登录 9222。

服务端身份上下文至少包含 `user_id`、`role` 和 `tenant_scope`。部门管理员的 `tenant_scope` 由服务端计算且只有一个 Tenant ID；超级管理员的范围为全局。

### 7.2 固定权限守卫

后端提供职责明确的权限守卫：

- `require_management_user`：允许两类管理角色。
- `require_super_admin`：只允许超级管理员。
- `require_department_scope`：校验资源是否处于当前用户允许的 Tenant 范围。
- `require_action(resource, action)`：执行固定权限矩阵中的动作校验。

现有只允许超级管理员的系统接口继续使用超级管理员守卫。新增知识库、聊天助手、聊天记录和 Channel 管理接口按权限矩阵分别使用守卫。

### 7.3 数据范围规则

- 部门管理员查询知识库时，后端强制使用身份上下文中的部门 `tenant_id`。
- 部门管理员创建知识库时，忽略请求中的 `tenant_id`，统一写入身份上下文的部门 ID。
- 部门管理员访问知识库或文档详情时，先校验其所属知识库的 `tenant_id`。
- 聊天助手通过自身 `tenant_id` 判断部门归属。
- Conversation 通过 `dialog_id -> Dialog.tenant_id` 判断部门归属。
- ChatChannel 通过自身 `tenant_id` 判断部门归属，但相关接口仅允许超级管理员。
- 超级管理员创建聊天助手时，所有 `kb_ids` 必须属于目标部门。
- 超级管理员连接 Channel 时，ChatChannel 与 Dialog 的 `tenant_id` 必须一致。

部门管理员无权查看的其他部门资源统一按 404 处理，避免泄露资源是否存在；已知资源但动作不允许时返回 403。

### 7.4 防绕过

部门管理员账号只允许使用 9222 管理门户。普通 RAGFlow 登录和业务接口需要识别部门管理员账号，拒绝其通过普通页面或直接请求绕过管理门户权限。

即使前端隐藏菜单，以下接口仍必须使用 `require_super_admin`：

- 聊天助手创建、修改、删除和测试。
- ChatChannel 创建、查看、修改、删除、启停和绑定。
- 模型、模型密钥和 API Token 管理。
- 聊天记录删除和保留策略修改。
- 部门及部门管理员管理。

## 8. 数据模型调整

### 8.1 Tenant

为 Tenant 增加部门类型标记，默认值兼容现有个人 Tenant。部门列表只查询部门类型的 Tenant。

部门停用时执行一致性操作：

- Tenant 状态置为无效。
- 部门管理员账号置为不可登录。
- 该部门所有 ChatChannel 状态置为停用。
- 知识库、聊天助手和聊天记录不删除，仍可由超级管理员查看和恢复。

### 8.2 UserTenant

扩展固定的部门管理员成员角色。一个部门管理员必须且只能有一条有效部门管理员成员关系。数据库迁移和服务层同时校验这一约束。

### 8.3 AuditLog

新增 `AdminAuditLog` 管理审计日志表，至少包含：

- 操作者用户 ID 和角色。
- 目标部门 Tenant ID。
- 动作、资源类型和资源 ID。
- 成功或失败结果。
- 请求 IP 和时间。
- 不包含 Secret、Token、密码和文档正文的安全详情。

以下动作必须记录审计：部门和管理员变更、知识库删除、聊天记录导出和删除、聊天助手变更、Channel 凭证或绑定变更、模型凭证变更。

## 9. 关键数据流

### 9.1 部门管理员管理知识库

```text
部门管理员登录 9222
  -> 服务端解析 department_admin 和唯一 tenant_scope
  -> 前端展示部门知识库页面
  -> 创建或修改知识库请求
  -> 后端固定使用 tenant_scope 中的 tenant_id
  -> KnowledgebaseService 执行业务操作
  -> 返回本部门数据
```

### 9.2 超级管理员创建聊天助手并绑定企微

```text
超级管理员选择目标部门
  -> 创建聊天助手
  -> 校验所有 kb_ids 均属于目标部门
  -> Dialog.tenant_id = 目标部门
  -> 创建企业微信 ChatChannel
  -> ChatChannel.tenant_id = 目标部门
  -> 校验 Dialog.tenant_id 与 ChatChannel.tenant_id 一致
  -> 保存 Bot ID / Secret 并启动 WebSocket
  -> 写入审计日志
```

### 9.3 部门管理员查看聊天记录

```text
部门管理员请求聊天记录
  -> 后端取得唯一部门 tenant_id
  -> 查询该部门 Dialog IDs
  -> 按 dialog_id 查询 Conversation
  -> 支持机器人、用户标识、关键词和时间筛选
  -> 返回记录或生成导出文件
  -> 导出操作写入审计日志
```

## 10. 现有数据迁移

升级后，现有知识库、聊天助手和 ChatChannel 不自动跨 Tenant 移动，避免破坏模型配置、对象存储地址和现有会话。

超级管理员在迁移页面执行以下操作：

1. 创建或标记部门 Tenant。
2. 将现有资源按当前 Tenant 整体认领为某个部门。
3. 校验每个聊天助手引用的知识库均处于同一部门。
4. 校验每个 ChatChannel 绑定的聊天助手与 Channel 属于同一部门。
5. 无法满足一致性的资源保持“未分配”，仅超级管理员可见且 Channel 暂停运行。
6. 全部资源通过校验后启用严格部门权限。

迁移必须可重复执行，并在任何失败时保留原数据。迁移过程不得记录或导出 Channel Secret。

## 11. 错误处理

- 未登录请求返回 401。
- 已登录但动作无权限返回 403。
- 部门管理员访问其他部门资源返回 404。
- 部门管理员没有有效部门、存在多个部门或部门停用时拒绝登录。
- 聊天助手绑定跨部门知识库时拒绝保存，并返回可理解的校验错误。
- Channel 绑定跨部门聊天助手时拒绝保存，不启动或重启连接。
- 部门停用时同步停止其 Channel；单个 Channel 停止失败不阻止其他状态更新，并记录错误。
- 聊天记录导出失败不生成不完整下载文件，并写入失败审计。
- 审计日志写入失败时，高风险写操作失败并回滚；只读查询可继续但记录系统错误。
- API 响应、应用日志和审计详情不得包含密码、Secret、Token 或完整模型密钥。

## 12. 测试设计

### 12.1 后端权限测试

- 超级管理员可以访问所有部门资源。
- 部门管理员只能列出本部门知识库、聊天助手和聊天记录。
- 部门管理员无法通过修改请求参数访问其他部门。
- 部门管理员无法调用聊天助手写接口和任何 Channel 接口。
- 其他部门资源详情返回 404，不允许动作返回 403。
- 部门管理员无法通过普通 RAGFlow 页面或接口绕过限制。
- 停用部门或管理员后立即禁止登录和后续请求。

### 12.2 数据一致性测试

- 聊天助手只能引用同部门知识库。
- Channel 只能绑定同部门聊天助手。
- Conversation 能够通过 Dialog 正确解析部门归属。
- 部门停用时 Channel 同步停用，业务数据不被删除。
- 模型用量能够归集到正确部门。

### 12.3 前端测试

- 两种角色显示正确菜单。
- 部门管理员看不到超级管理员按钮和敏感字段。
- 直接访问无权限路由时展示 403。
- 部门管理员页面不允许编辑 `tenant_id`。
- 超级管理员创建资源时必须选择目标部门。
- 列表筛选、聊天记录导出和空状态行为正确。

### 12.4 审计测试

- 知识库删除、聊天记录导出和删除、聊天助手变更、Channel 变更均生成审计记录。
- 审计记录包含操作者、角色、部门、资源和结果。
- 审计记录不包含任何 Secret、Token 或密码。

### 12.5 迁移测试

- 现有个人 Tenant 不会被误识别为部门。
- 资源认领成功后归属正确。
- 跨部门引用会阻止迁移完成。
- 迁移失败可安全重试，原数据和现有凭证不丢失。

## 13. 验收标准

1. 9222 Admin 页面同时支持超级管理员和部门管理员登录。
2. 超级管理员可以在一个门户管理所有部门和业务资源。
3. 部门管理员只能管理本部门知识库。
4. 部门管理员只能只读查看本部门聊天助手。
5. 部门管理员可以查看、搜索和导出本部门聊天记录，但不能删除。
6. 部门管理员无法通过前端地址或直接 API 请求访问其他部门、创建聊天助手或管理 Channel。
7. 聊天助手与知识库、Channel 与聊天助手之间不存在跨部门绑定。
8. 普通员工无需后台账号，可继续通过企业微信问答。
9. 关键高风险操作生成不包含敏感凭证的审计日志。
10. 现有资源能够安全完成部门认领，未分配资源仅超级管理员可见且不会继续运行 Channel。

## 14. 实施顺序建议

1. 数据库字段、固定角色和部门 Tenant 约束。
2. Admin 登录身份上下文和后端权限守卫。
3. 部门、知识库、聊天助手、聊天记录和 Channel 管理 API。
4. 9222 统一门户菜单、路由和业务页面。
5. 审计日志和聊天记录导出。
6. 现有数据迁移和一致性校验。
7. 权限、跨部门、防绕过和端到端测试。
